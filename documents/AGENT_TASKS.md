## Backend Contract Target

These are target backend designs for the current batch. If current live backend lacks an endpoint, build frontend code with clear error/fallback states in `REMOTE`; do not fake success.

### Debt List

Existing endpoint:

- `GET /api/v1/loan/debts`
- Auth: required
- Query:
  - `include_paid`: optional boolean, default `false`

Target behavior:

- Default response excludes debts where `status == PAID`.
- Frontend must also filter out `PAID` debts defensively until backend is deployed.
- Paid debts may be shown only in a dedicated history view if product adds one later.

### Reward Points

Reward points are spendable loyalty points. They are not credit score.

Events:

- Loan disbursement adds reward points.
- Completing a debt, meaning the debt status becomes `PAID`, adds reward points.
- Ordinary monthly repayment may show a preview but should not be the only source of large rewards unless backend explicitly returns it.

Relevant endpoints:

- `GET /api/v1/rewards/user`
- `POST /api/v1/loan/contracts/{contract_id}/sign`
- `POST /api/v1/loan/debts/{debt_id}/repay`

Disbursement/sign success may return:

```json
{
  "contract_id": "CONTRACT_APP_XXXXXXXX",
  "application_id": "APP_XXXXXXXX",
  "status": "DISBURSED",
  "debt_id": 12,
  "disbursed_amount": 20000000,
  "reward_points_added": 100,
  "total_reward_points": 1600,
  "credit_score_delta": 10,
  "credit_score": 720
}
```

Repay success may return:

```json
{
  "debt_id": 12,
  "repay_type": "FULL_EARLY",
  "amount_paid": 11760000,
  "reward_points_added": 120,
  "total_reward_points": 1720,
  "credit_score_delta": 30,
  "credit_score": 750,
  "debt_status": "PAID"
}
```

Frontend requirement: after disbursement or any repay success, refresh rewards/profile state where visible. Do not add credit-score refresh because frontend does not display user credit score.

### Credit Score

Credit score is a backend-owned underwriting score used for loan underwriting. Frontend does not display user credit score in the current product scope.

Backend endpoint, documented for context only:

- `GET /api/v1/user/credit-score`
- Auth: required
- Response: `ApiResponse[CreditScoreDto]`

```json
{
  "credit_score": 720,
  "tier": "GOOD",
  "min_score": 0,
  "max_score": 1000,
  "total_reward_points": 1347,
  "explanation": "Điểm tín dụng dùng để xét điều kiện vay; điểm thưởng dùng riêng cho ưu đãi và voucher."
}
```

Backend policy target, for frontend context only:

- Base score for a new verified user: around `620`.
- Incomplete eKYC starts lower, around `520`.
- Successful disbursement may add a small positive delta.
- Monthly on-time repayment may add a small delta.
- Completing a debt adds a larger positive delta.
- Rejected applications, overdue debt, or failed repayment may reduce score.
- Loan eligibility is evaluated by backend and exposed through loan package/eligibility APIs.

Frontend requirement:

- Do not add `CreditScoreDto`, `CreditScoreModel`, `getCreditScore()`, repository methods, ViewModels, or UI for `/api/v1/user/credit-score` in this task batch.
- Do not call `/api/v1/user/credit-score` from frontend unless a future approved design adds a visible credit-score screen.
- Do not compute credit score locally from reward points, profile completion, loan package thresholds, or repayment history.
- Existing loan eligibility UI should continue to trust backend loan APIs (`is_eligible`, `reason_code`, package thresholds/messages). Frontend must not make its own underwriting decision.
- Sign/repay responses may include `credit_score_delta` and `credit_score`; frontend may ignore these fields unless they are needed for a visible success message. They must not be used as reward points.

### Bank Card Add/Verify

Metadata endpoint:

- `GET /api/v1/payment/banks`
- Response: `ApiResponse[List[BankDto]]`

```json
{
  "id": "VCB",
  "name": "Vietcombank",
  "short_name": "VCB",
  "bin_prefixes": ["970436"],
  "logo_url": null,
  "supported_card_types": ["DEBIT", "CREDIT"]
}
```

Verify/add endpoints:

- `POST /api/v1/payment/cards/verify`
- `POST /api/v1/payment/cards`

Canonical request. Frontend must send this exact shape:

```json
{
  "bank_id": "VCB",
  "bank_name": "Vietcombank",
  "card_type": "DEBIT",
  "card_number": "9704361234567890",
  "card_holder_name": "NGUYEN VAN A",
  "expiry": "10/2026",
  "cvv": "123"
}
```

Rules:

- `expiry` is a string in `MM/YYYY` format. Month `10`, year `2026` must be sent as `10/2026`.
- Frontend may use a date picker or masked text field, but must normalize to `MM/YYYY` before calling backend.
- `cvv` is a string. It may be labelled CVV or CVC in UI, but request key is `cvv`.
- Card number should be sent as digits only after removing spaces.
- Show inline validation and preserve form state on error.

Structured error shape:

```json
{
  "status": "error",
  "code": "INVALID_CARD_EXPIRY",
  "message": "Thong tin the khong hop le.",
  "field_errors": {
    "expiry": "Ngay het han phai dung dinh dang MM/YYYY."
  }
}
```

### Contract OTP

OTP request endpoint:

- `POST /api/v1/loan/contracts/{contract_id}/sign/request-otp`
- Auth: required
- Response: `ApiResponse[ContractOtpRequestResultDto]`

```json
{
  "contract_id": "CONTRACT_APP_XXXXXXXX",
  "purpose": "SIGN_CONTRACT",
  "otp": "123456",
  "expires_at": 1717240300000
}
```

Expected FCM data payload:

```json
{
  "type": "CONTRACT_SIGN_OTP",
  "purpose": "SIGN_CONTRACT",
  "contract_id": "CONTRACT_APP_XXXXXXXX",
  "otp": "123456",
  "expires_at": "1717240300000",
  "title": "Ma OTP ky hop dong",
  "body": "Ma OTP ky hop dong vay EasyMoney cua ban la 123456. Ma co hieu luc trong 5 phut."
}
```

Final signing endpoint:

- `POST /api/v1/loan/contracts/{contract_id}/sign`
- Auth: required
- Request: `ContractSignRequest`

```json
{
  "otp": "123456",
  "purpose": "SIGN_CONTRACT"
}
```

Frontend requirement:

- Request OTP.
- Do not automatically fill the OTP field as soon as the API/FCM OTP arrives.
- Offer the matching OTP as a user-selected autofill option. Prefer Android keyboard/autofill suggestion when feasible; an in-app suggestion chip directly above the OTP field is acceptable if keyboard integration is not reliable for FCM data messages.
- The user may still type OTP manually.
- Do not auto-submit; user still taps the final confirm button.
- Ignore OTP payloads whose `contract_id` or `purpose` does not match the active signing flow.
- Track request state per active `contract_id` in frontend state. After OTP has already been requested and has not expired, reopening or returning to the signing UI must show the existing OTP entry/suggestion UI instead of calling `request-otp` again.
- Only call `request-otp` again when the user explicitly asks to resend after expiry or an error.

```json
  "body": "Khoan vay da duoc giai ngan vao vi EasyMoney.",
  "target_type": "LOAN_DEBT",
  "target_id": "12"
}
```

Frontend requirement: notification tap navigates to loan management for the target debt when possible.

### Notification User Isolation

Existing endpoint:

- `GET /api/v1/notifications`
- Auth: required

Target backend behavior:

- Response contains only rows where `notification.user_id == authenticated_user.id`.
- A newly registered user should receive `data: []` unless backend has created notifications for that exact user.
- Mark-read, read-all, and clear actions must also be scoped to the authenticated user.

Frontend requirement:

- Do not display cached notifications for a different account.
- Room notification rows must be keyed/queried by the current authenticated user id.
- After login/register/account switch, refresh notifications for the new user and clear or ignore stale rows from the previous user.
- FCM payloads saved locally must use the current authenticated user id, not a hard-coded seed id such as `user_123` or `1`.

### 8. Notification Cache Isolation After Register/Login

- Audit notification repository, DAO queries, Room entity mapping, ViewModel state, and `EasyMoneyMessagingService`.
- Remove any hard-coded user id from notification cache paths.
- Derive current user id from `AppPreferences`/token/session consistently.
- On register/login/account switch, do not reuse old notification UI state for the new user.
- In `REMOTE`, an empty backend notification list for a new user is valid and must render an empty state.
- Ensure read/clear operations call backend for the current user and only mutate local rows for that user.

Acceptance:

- Register a new account, open notifications, and see no seed notifications from user `1`.
- Log out/login between seed user and new user; notifications do not leak between accounts.
- FCM-created local notifications are stored under the current user id.

Verified frontend bug details:

- API call/auth are not the primary problem:
  - `data/remote/LoanApiService.kt` calls `GET api/v1/notifications`.
  - `data/remote/AuthInterceptor.kt` attaches `Authorization: Bearer <accessToken>` for non-auth endpoints.
  - Backend quick check with a newly registered token returned `data: []`, so user leakage is very likely frontend cache/state.
  - Backend also had an auto-promotion startup job that could create default package-offer notifications for every existing user; backend has removed that job. Frontend still must handle empty backend lists and account switches correctly.
- `ui/notification/NotificationViewModel.kt` captures the user id once:

```kotlin
private val currentUserId: String = appPreferences.currentUserId
```

  If user 1 opens Notifications, then the app logs out/registers/logs in as another user while the ViewModel/backstack remains alive, the screen can still observe `user_1` cache.
- `data/local/dao/NotificationDao.kt` reads are user-scoped, but mutations are not:
  - `markAsRead(id)` updates by `id` only.
  - `markAllAsRead()` updates the whole `notifications` table.
  - `clearAll()` deletes the whole `notifications` table.
- `data/local/entity/NotificationEntity.kt` uses backend `id` as the sole Room primary key:

```kotlin
@PrimaryKey(autoGenerate = true)
val id: Long = 0
```

  `domain/repository/NotificationRepositoryImpl.kt` then writes `id = dto.id`. If backend notification ids overlap across users, `OnConflictStrategy.REPLACE` can overwrite cache rows between accounts. The local key must be user-namespaced or separated from backend `remoteId`.
- `domain/repository/NotificationRepositoryImpl.kt` inserts refreshed backend rows but does not replace the current user's cache with the exact backend result. If backend returns an empty list for a new user, stale rows for that user id should be cleared/reconciled.

Required fix direction:

- Do not store `currentUserId` as an immutable field inside `NotificationViewModel`. Either:
  - derive it from `AppPreferences.currentUserId` every time a flow/action is built, or
  - expose a user-id flow/session flow and `flatMapLatest` notification queries by current user.
- Make DAO mutations user-scoped:
  - `markAsRead(id, userId)`
  - `markAllAsRead(userId)`
  - `clearAll(userId)`
  - optionally `deleteForUser(userId)` / `replaceForUser(userId, rows)` for refresh reconciliation.
- Change local persistence so notification rows cannot collide across users:
  - preferred: add `localId: String = "${userId}_${remoteId}"` as `@PrimaryKey`, and keep backend id as `remoteId: Long`;
  - acceptable: use a composite-equivalent stable key if Room entity design already supports it.
- Update repository methods to pass the current user id into DAO mutation calls.
- In `refreshNotifications()`, reconcile the current user's cache with backend result. When backend returns `[]`, current user's notification rows should become empty.
- Ensure `EasyMoneyMessagingService` stores FCM-created local notifications under the current authenticated user id. If no authenticated user exists, either ignore the payload or store it under a non-production anonymous bucket that production notification screen never queries.
- Consider clearing notification UI state/cache on successful login/register/account switch, but do not rely on global table delete as the only fix.
